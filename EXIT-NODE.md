# Exit node — VPN data plane (feature `vpn`)

This document describes how to **actually** route a device's traffic through
another one marked as an *exit node*, using the data plane implemented in
`src/vpn.rs` (feature `vpn`).

> ⚠️ **Honest status.** The data plane is implemented and **compiles**. Covered
> by **unit tests** (`cargo test --features vpn`): framing/demux, IPAM, IPv4
> parsing and — most importantly — the **WireGuard handshake with an encrypted
> round-trip** (two `Tunn` sessions exchanging an encrypted packet that decrypts
> correctly). Actually routing traffic to the Internet, however, **has not been
> verified**: opening the TUN requires **root** privileges (without them it fails
> with `Operation not permitted`) and the exit should run on a suitable host
> (typically **Linux**). Treat it as a **prototype with verified crypto but
> end-to-end routing still to be validated in the field**.
>
> ✅ **Encryption.** The tunnel uses **WireGuard** (boringtun): each client↔exit
> pair performs a Noise handshake and encrypts packets (ChaCha20-Poly1305). The
> X25519 keys are generated by the client on first run and distributed via the mesh.

## How it works

Virtual subnet `10.7.0.0/24`, with per-device virtual IPs assigned by the
control plane (IPAM):

- **Exit node** → TUN on its assigned virtual IP, with IP forwarding + NAT
  masquerade to the Internet. It receives the clients' packets on the UDP socket
  already "punched" by the mesh, decrypts them, writes them to its TUN (the
  kernel routes and NATs them), and sends the replies back to the right client
  (a *virtual IP → UDP endpoint* map, learned from traffic).
- **Client** → TUN on its assigned virtual IP. Local packets enter the TUN, get
  WireGuard-encrypted, and are sent to the exit node over the punched socket.

The UDP socket is the **same** one used by the mesh: VPN datagrams are told apart
from hole-punch PINGs by a magic byte, and the client's receiver dispatches them
to the tunnel (demux in `spawn_receiver`).

## Build

```bash
# Client + data plane
cargo build --release --features vpn --bin client
```

## Usage

Prerequisites: the devices are already logged in (device-code flow) and the exit
node is marked as such in the backoffice (**Devices → Rendi exit node**).

### 1. On the exit node (ideally a Linux VPS)

```bash
sudo ./client --exit-node
```

On startup it tries to configure itself, on Linux:

```
sysctl -w net.ipv4.ip_forward=1
iptables -t nat -A POSTROUTING -s 10.7.0.0/24 -j MASQUERADE
```

If you prefer to do it by hand (or run a kernel with a managed firewall), run the
same commands before starting. On **macOS** the auto-config is not supported; use
`pf`:

```
sysctl -w net.inet.ip.forwarding=1
echo 'nat on en0 from 10.7.0.0/24 to any -> (en0)' | sudo pfctl -f - -e
```

### 2. On the client that wants to exit through the exit node

```bash
sudo ./client --use-exit "exit-node-name"
```

As soon as the exit node is online in the mesh, the client sends `UseExitNode`,
gets confirmation from the server, and starts the tunnel toward the exit's
punched endpoint.

### 3. Traffic routing (manual, for now)

Bringing the TUN "up" with an address automatically adds the route for the
`10.7.0.0/24` subnet. To route **all** traffic you need, on the client, the
classic VPN routes (Linux), to be adapted:

```bash
# Keep the exit node reachable via its normal path
sudo ip route add <EXIT_PUBLIC_IP>/32 via <CURRENT_GATEWAY>
# Send everything else into the TUN (0/1 + 128/1 = default without overriding it)
sudo ip route add 0.0.0.0/1 dev tun0
sudo ip route add 128.0.0.0/1 dev tun0
```

These routes are **not** set automatically on purpose: manipulating the default
route is delicate and must be done carefully (risk of locking yourself out of the
network). For a safer test, route only a specific subnet into the TUN instead of
everything.

## Recommended verification (in the field)

1. Two Linux hosts (or a Linux client + a Linux VPS as exit).
2. Start the server, log in the devices, mark the exit.
3. `sudo ./client --exit-node` on the exit, `sudo ./client --use-exit <name>` on
   the client, then the routes from step 3.
4. From the client: `curl -s https://ifconfig.me` → it must show the **exit
   node's public IP**. That is the exit node "working".

## What is already implemented

- **WireGuard encryption** (boringtun) end-to-end between client and exit node.
- **IPAM**: the server assigns stable, distinct virtual IPs (`10.7.0.x`), saved
  on the device and distributed via the mesh.
- **Exit-node selection from the client**: CLI (`--use-exit <name>`) and the
  Tauri app (dropdown of the available exit nodes).

## Known limitations of this cut

- **Routing not verified**: the TUN requires root; full-tunnel routing is manual
  (commands above) and must be tested on a real host.
- **IPv4 only**. No automatic DNS push.
- **Symmetric NAT** on the mesh side: if direct hole punching fails, a relay
  (DERP-style) is needed, not present yet.
